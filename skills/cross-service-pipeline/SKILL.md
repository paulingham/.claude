# Cross-Service Pipeline

## What This Skill Does

Coordinates changes that affect multiple services or repos. Verifies contract compatibility, generates deployment plans, runs cross-service contract tests, and manages backwards compatibility windows. This skill orchestrates — it doesn't replace per-service pipelines.

## When to Invoke

- A change modifies a shared contract (OpenAPI spec, Protobuf, event schema)
- A change requires coordinated deployment across services
- Adding a new service that depends on existing services
- Removing or deprecating an API endpoint consumed by other services

## Prerequisites

- Project CLAUDE.md has a `## Service Context` section (generated by `/project-setup`)
- Shared contracts are accessible (in a shared-contracts repo or co-located)

## Process

### Step 1: Map Impact

Identify all affected services:

```markdown
## Impact Map

### Changed Contract
- File: [path to OpenAPI/Protobuf/event schema]
- Change type: [additive / breaking]

### Provider Service
- Repo: [provider repo]
- Endpoint/topic affected: [specific endpoint or event topic]

### Consumer Services
- [consumer-1]: uses [endpoint] for [purpose]
- [consumer-2]: uses [endpoint] for [purpose]

### Impact Classification
- [ ] Additive (new field, new endpoint) → safe to deploy provider first
- [ ] Breaking (removed field, changed type) → requires versioning + migration window
```

### Step 2: Verify Contract Compatibility

Run compatibility checks on the changed contract:

**OpenAPI:**
```bash
# Compare old and new spec for breaking changes
npx openapi-diff old-spec.yaml new-spec.yaml
# Or: oasdiff breaking old-spec.yaml new-spec.yaml
```

**Protobuf:**
```bash
# Check backward compatibility
buf breaking --against .git#branch=main
```

**Event Schema (JSON Schema):**
```bash
# Validate new schema is backward compatible
# New schema must accept all data valid under old schema
```

Classify result:
- **Compatible**: no breaking changes detected → proceed with deployment
- **Breaking**: breaking changes detected → require versioning plan (Step 3b)

### Step 3a: Deployment Plan (Compatible Changes)

```markdown
## Deployment Plan (Additive Change)

### Order
1. Deploy provider with new field/endpoint (backward compatible)
2. Consumers can update on their own timeline
3. No coordination required

### Verification
- Provider deploys → health check passes
- Provider's contract tests pass
- Consumer contract tests still pass (additive = backward compatible)
```

### Step 3b: Deployment Plan (Breaking Changes)

```markdown
## Deployment Plan (Breaking Change)

### Versioning Strategy
- New version: /api/v2/[resource] (or new event schema version)
- Old version: /api/v1/[resource] remains active during migration

### Migration Timeline
- Week 1: Deploy provider with v2 (v1 still active)
- Week 2: Consumer PRs to migrate to v2
- Week 3: Monitor v1 usage → should reach zero
- Week 4: Remove v1 from provider

### Deploy Order
1. Provider: deploy with both v1 and v2 active
2. Consumer-1: PR to use v2 → deploy
3. Consumer-2: PR to use v2 → deploy
4. Provider: remove v1 after all consumers migrated

### Sunset Headers
Add to v1 responses:
  Sunset: [date 4 weeks from now]
  Deprecation: true
```

### Step 4: Run Contract Tests

```bash
# Consumer-driven contract tests
# Each consumer has published its expectations to the contract broker

# Provider verification: does the new provider satisfy all consumer contracts?
pact verify --provider-base-url=http://localhost:3000 --pact-broker-url=https://pact-broker.internal

# Or with Specmatic:
specmatic test --contract=openapi.yaml --host=http://localhost:3000
```

If any consumer contract fails: **CROSS_SERVICE_BLOCKED** — the change breaks a consumer's expectations. Fix the provider or coordinate with the consumer team.

### Step 5: Generate CI/CD Coordination

For changes requiring coordinated deployment:

```yaml
# Trigger downstream CI when contract changes
# .github/workflows/contract-change.yml
on:
  push:
    paths: ['contracts/**']

jobs:
  notify-consumers:
    runs-on: ubuntu-latest
    steps:
      - uses: peter-evans/repository-dispatch@v2
        with:
          token: ${{ secrets.CROSS_REPO_TOKEN }}
          repository: org/consumer-service-1
          event-type: contract-updated
          client-payload: '{"contract": "task-api", "version": "2.0.0"}'
```

### Step 6: Post-Deploy Cross-Service Verification

After all services are deployed:

```markdown
## Cross-Service Smoke Tests
- [ ] Service A → Service B: [specific API call] returns expected data
- [ ] Event flow: publish event on A → consumer B receives and processes
- [ ] End-to-end: [user action] → traverses all affected services → correct result
- [ ] No increase in cross-service error rates (monitor for 15 minutes)
```

## Phase Output

```
Verdict: CROSS_SERVICE_VERIFIED / CROSS_SERVICE_BLOCKED
Next: Execute deployment plan / Fix contract incompatibility
Artifacts: [impact map, compatibility report, deployment plan, contract test results]
Breaking: [yes/no]
Migration Window: [N weeks, or N/A if compatible]
Affected Services: [list]
```
