# Baseline Context — 2026-04-24 (Opus)

## Result
`0/1 = 0%` — single case `per-project-instincts-bootstrap-pr19` returned `failed_build`.

## Root Cause
The inner `claude -p "/pipeline ..."` CLI invocation is **single-shot**. It
prints a response and exits. When the inner responds with a clarifying
question (rather than executing the full `/pipeline` skill), `claude -p`
exits 0 — but the pipeline never actually ran.

This run's inner stdout showed exactly that pattern:

> "Before spawning the pipeline I need to flag a context issue for you...
> Three options: 1. Run pipeline off `main`... 2. Stack on current branch...
> 3. Cancel... Which do you want?"

The inner asked the operator to choose a branching strategy, then exited 0.
The short-circuit guard (`_real_postcheck` requires `PIPELINE_COMPLETE` in
stdout) correctly flipped this to `failed_build`.

## Why This Is Honest, Not Broken

The current harness architecture assumes the inner `/pipeline` can run to
completion in a single non-interactive invocation. That assumption does not
hold: multi-turn pipelines require a session (not `claude -p`). Until the
harness wraps the inner in a session-capable driver, or teaches the inner
to execute `/pipeline` end-to-end without clarifying questions, this
baseline will stay at 0%.

This is the correct number to publish. A 100% baseline captured under the
previous stub-leaking dispatcher (status=passed without the inner actually
building anything) was a fake positive and has been removed.

## What The Baseline Does Prove

- Harness dispatch path works end-to-end (run-suite → run-case → real
  `claude -p` → postcheck → aggregate → baseline write).
- Isolation env contract is exported to the real inner (bypass, disable-
  auto-learn, eval ids).
- Short-circuit detection catches inner-asks-question vs inner-completes
  the pipeline. Without this guard, the baseline would silently report
  100% regardless of what the inner actually did.

## Next Steps (Not Part Of This PR)

Either:
1. Replace `claude -p` with a session-capable driver that can keep the
   inner turning until `/pipeline` completes, OR
2. Teach the inner to execute `/pipeline` autonomously (no clarifying
   questions) when `CLAUDE_PIPELINE_BYPASS=1` and `EVAL_RUN_ID` are set.

Until one of those lands, publishing a real pass rate requires a suite of
cases whose `/pipeline` runs do not ask questions — or a session driver.
